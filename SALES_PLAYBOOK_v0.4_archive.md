## v0.3 → v0.4 結構升級

Livia 給的 4 段 pipeline：

```
[1] 進案路徑（觸發點 + 窗口/有決策權的人）
        ↓
[2] 依客戶 profile 走決策樹
        ↓
[3] AI at Scale / Governance 可包的專案組合（outcome）
        ↓
[4] 依客戶 profile 走決策樹決定銷售包裝
        - 緣起和痛點
        - 用啥方式賣解決方案（= 售賣 Frame）
        - 整體語氣和 style
```

v0.4 對應 v0.3 的變動：

| v0.3 | v0.4 |
|---|---|
| Part B 進案路徑（只有觸發點）| **Part B 進案路徑（觸發點 × 窗口）**——新增窗口維度 |
| Part C 決策樹（直接訊號 → outcome）| **Part C 客戶 profile → outcome 決策樹**——加客戶 profile 概念層 |
| Part D 包裝層（4 維度 + 5 archetype）| **Part D 包裝三件事**：D1 緣起痛點 / D2 售賣 Frame / D3 語氣 Style |

**最大新增：售賣 Frame**——「IBM 在這個案子扮演什麼角色的核心定位敘事」，跟 Style（怎麼說話）是兩件事。Livia 例子：「都是賣治理全套，工具思維 vs 顧問思維」就是兩個 Frame。

# Part A：可賣結果套餐 catalog（v0.3 沿用）

## A1. AI at Scale 套餐（4 種，Livia named）

### AS-1 「先場景後診斷」

- **Livia 原話**：只先賣場景設計和落地，剩下真診斷、來年再加強
- **賣的東西**：今年場景 1-2 個，明年顧問藍圖
- **模組組合**：#6 ×1-2 + #3 D2 minimal + 後續 #1 hooks
- **規模**：今年 50-150 人月、6-12 個月
- **Sales motion**：兩段 play（小 → 擴大）

### AS-2 「純顧問藍圖」

- **Livia 原話**：做顧問案
- **賣的東西**：純顧問藍圖，不落地
- **模組組合**：#1 B1-B5 完整 + 部分 #2 + #5 M1 + #4 T1-T2
- **規模**：100-400 人月、6-12 個月
- **Sales motion**：純顧問交付

### AS-3 「顧問 + 場景 + 落地一案到底」

- **Livia 原話**：做顧問案 + 挑場景 + 真的落地
- **賣的東西**：顧問 + 同案內挑場景 + 落地
- **模組組合**：#1 + #6 ×N + #3 + 部分 #4 + 部分 #5 + 部分 #2 + 部分 #7
- **規模**：500-1500 人月、12-24 個月
- **Sales motion**：旗艦級

### AS-4 「純場景」

- **Livia 原話**：只做場景
- **賣的東西**：1-N 個場景，無顧問
- **模組組合**：#6 ×1-3 + #3 D2 minimal + #4 T1 minimal
- **規模**：30-200 人月、3-12 個月
- **Sales motion**：直接進場景案

## A2. AI Governance 套餐（5 種，Livia 截圖 ASSET MAP）

### GV-A 補強升級包（高成熟）
- 客戶現況：已有完整 AI 內規、已實施
- 主打模組：M3 機制補強 + M5 工具升級
- 規模：50-150 人月、3-9 個月

### GV-B 基於內規重構 + 落地包（中成熟）
- 客戶現況：有零散內規但寫不好、有做過一次性檢測
- 主打模組：基於 M4 既有內規，實施 M1-M9
- 規模：200-500 人月、9-18 個月

### GV-C 全套新建 + 落地包（低成熟）
- 客戶現況：沒內規從零開始
- 主打模組：實施 M1-M9（全）
- 規模：300-800 人月、12-24 個月

### GV-D 治理工具棧（工具型）
- 客戶現況：只想買工具
- 主打模組：M5 為主
- 規模：50-150 人月、3-6 個月

### GV-E 一次性技術檢測（不算治理）
- 客戶現況：只想針對某專案表現進行驗證
- 主打模組：M8-M9 技術控制項
- 規模：30-80 人月、1-3 個月
- ⚠️ Corner case：若內規不清楚 → 回到 GV-B 或 GV-C

⚠️ Livia 截圖標題「**支柱一·對外 GTM**」——目前只討論支柱一。

# Part C：客戶 profile → outcome 決策樹

## C1. 客戶 profile 維度（10 維，Claude 整理）

從進案路徑帶來的線索 + 後續蒐集，組成完整客戶 profile：

| # | 維度 | 軸 |
|---|---|---|
| 1 | **AI at Scale 應用成熟度**（接 wiki 已有的 T1-T5 / M1-M5）| 低 / 中 / 高 |
| 2 | **AI 治理成熟度**（Livia 截圖 A/B/C/D/E 對應）| 高 / 中 / 低 / 工具型 / 單案型 |
| 3 | **預算量級** | < 3k 萬 / 3k 萬-2 億 / > 2 億 |
| 4 | **時程要求** | < 6 個月 / 6-18 個月 / > 18 個月 |
| 5 | **決策深度** | CEO 直線 / CDO 主導 / CIO 落地 / 業務 BU |
| 6 | **集團結構** | 單體 / 母子分散 / 集團整合中 / 跨國 |
| 7 | **文化** | 保守 / 中 / 激進 |
| 8 | **法規緊迫度** | 強（被盯）/ 中 / 弱 |
| 9 | **既有 IBM 關係** | 有大案 / 有 AM / 弱 / 無 |
| 10 | **Frontier sophistication** | CEO 讀 / CDO 讀 / IT 讀 / 無人讀 |

## C2. AI at Scale 決策樹

```
[Step 1] 客戶要藍圖嗎？（profile 維度 5 + 1）

├─ 要 + 要落地 → AS-3（顧問+場景+落地一案到底）
├─ 要 + 不要落地 → AS-2（純顧問藍圖）
└─ 不要藍圖 → 進 Step 2

[Step 2] 客戶有「後續擴大」意願嗎？（profile 維度 1 + 5）

├─ 有（明年要做藍圖）→ AS-1（先場景後診斷）
└─ 沒有（純就要場景）→ AS-4（純場景）
```

關鍵 profile 訊號：
- AS-3 適配：成熟度中-高、預算 > 2 億、時程 > 18 個月、決策深度 CEO/CDO
- AS-2 適配：成熟度中-高、預算彈性、時程中、決策深度 CEO/董事會
- AS-1 適配：成熟度中、預算中、時程中、決策深度 CDO
- AS-4 適配：成熟度低-中、預算 < 8k 萬、時程短、決策深度 CDO/CIO/業務

## C3. AI Governance 決策樹

```
[Step 1] 客戶要不要做整體治理？（profile 維度 2 + 8）

├─ 不要，只想買工具 → GV-D 治理工具棧
├─ 不要治理整體，只要單案技術檢測 → GV-E
│   └─ ⚠️ 若客戶內規不清 → 回到 GV-B 或 GV-C
└─ 要做整體治理 → 進 Step 2

[Step 2] 客戶治理成熟度？（profile 維度 2）

├─ 高成熟（已有完整內規、已實施）→ GV-A 補強升級包
├─ 中成熟（零散內規寫不好）→ GV-B 基於內規重構 + 落地
└─ 低成熟（沒內規）→ GV-C 全套新建 + 落地
```

## C4. AS × GV outcome 配對 bundle

| Bundle | AS outcome | GV outcome | 典型 profile |
|---|---|---|---|
| 「速贏」 | AS-4 純場景 | GV-E 技術檢測 | 預算薄 / 成熟度低 |
| 「先場景再藍圖」 | AS-1 先場景後診斷 | GV-D 工具棧 | 中型想長期 |
| 「中型一案到底」 | AS-3（中規模）| GV-B 中成熟內規重構 | 中型有基礎 |
| 「旗艦」 | AS-3（大規模）| GV-C 全套新建 | 大型 / CTBC 型 |
| 「治理 only」 | （無）| GV-B / GV-C | 玉山型法規驅動 |

| 「危機後」 | （無 或 AS-4 補洞）| GV-A 補強 / GV-B 補洞 | 剛出事的 |

# Part D：包裝三件事（依客戶 profile 走決策樹）

> 性質：Outcome 決定範圍與模組（賣什麼）。包裝決定對這個客戶怎麼講。Livia 拆三件事：**緣起痛點 / 售賣 Frame / 語氣 Style**。

## D1. 緣起與痛點（開場故事 hook）

> 性質：每個觸發點對應一個天然的緣起 hook。第一張 deck slide 通常就是這個。

| 觸發點 | 緣起 hook（範例措辭）|
|---|---|
| T1 高層自發轉型 | 「3 年後您想跟 frontier 對齊嗎？/ 競爭對手 X 已經到 T3」|
| T2 法規驅動 | 「金管會去年罰了 X 千萬，AI 治理是下一波重點 / EU AI Act 高風險清單」|
| T3 危機事件後 | 「您看 X 公司上週 AI 事件嗎？/ 您內部 incident 的根因是 ...」|
| T4 IBM 既有大案延伸 | 「您現在的 IBM 案治理控制項可以一鍵升級到 ...」|
| T5 watsonx 商機 | 「您買 watsonx 自然有 watsonx.governance ... 」|
| T6 三朵雲合作 | 「雲商不擅長的合規 mapping，IBM 可以補」|
| T7 M&A / 集團整合 | 「集團併購後 12 個 BU 各做各的 ...」|
| T8 認證取證 | 「ISO 42001 是您出海前必拿的 ...」|

額外的「業務痛點」hook（與觸發點正交，更具體）：
- 「您客服 AI 命中率還停在 40% 嗎？」（業務 BU 痛點）
- 「您 IT 預算 30% 花在重複資料整備嗎？」（IT 痛點）
- 「您理專每天 60% 時間在找資料嗎？」（業務生產力）

## D2. 售賣 Frame（IBM 在案子裡是誰）

> ⭐ **v0.4 新增層**。從 Livia 例子萃取：「都是賣治理全套，工具思維 vs 顧問思維」——這是 Frame 不同。Frame 跟 Style（語氣）是兩件事，跟 outcome（範圍）也是兩件事。

### Frame 的概念

**Frame = 客戶心裡覺得買的是什麼 + IBM 在案子裡扮演的角色**。同一個 outcome 可走不同 Frame；同一個客戶不同窗口可吃不同 Frame。

### 9 個售賣 Frame（Claude 推）

#### F1 工具供應商 frame
- **核心敘事**：「IBM 工具裡都 built-in 了，導工具就拿到治理基線」
- **客戶心裡買的**：工具 + 自帶治理 + 廠商品質背書
- **適配 outcome**：GV-D / GV-A 補強 / AS-3 中的 #4 平台部分
- **適配窗口**：W4 CIO / IT 主管 / 採購
- **適配觸發點**：T4 IBM 大案 / T5 watsonx / T6 三朵雲
- **失效模式**：客戶覺得工具貴 / 內規不清工具空轉 / 被開源打掉

#### F2 顧問藍圖 frame
- **核心敘事**：「IBM 顧問幫你想清楚 3 年怎麼做」
- **客戶心裡買的**：戰略清晰度 + 執行 roadmap + 第三方 validate
- **適配 outcome**：AS-2 / AS-3 / GV-C
- **適配窗口**：W1 CEO / W2 CDO / W9 董事會
- **適配觸發點**：T1 高層自發 / T7 M&A
- **失效模式**：MBB 故事更好 / 客戶覺得「我自己也能想」/ 純顧問落地不了

#### F3 標準服務內建 frame
- **核心敘事**：「治理是 IBM 案子的標準交付，不用額外買」
- **客戶心裡買的**：「IBM 案 = 治理 ready」+ 隱性品質
- **適配 outcome**：GV-D / GV-E / GV-A 大案 piggyback
- **適配窗口**：W4 CIO / W2 CDO（買主案的人）
- **適配觸發點**：T4 IBM 大案 / T5 watsonx
- **失效模式**：客戶把治理當「免費贈品」殺價時先砍 / 主案被砍治理跟著消失

#### F4 平台選型專家 frame
- **核心敘事**：「我們幫你選對平台組合（IBM / 開源 / 三朵雲）」
- **客戶心裡買的**：選型專業 + 整合能力 + 中立性
- **適配 outcome**：AS-3 中的 #4 / GV-D
- **適配窗口**：W4 CIO / W2 CDO
- **適配觸發點**：T1 平台升級 / T6 三朵雲決策
- **失效模式**：客戶覺得 IBM 不可能中立 / 雲商 / SI 直接競爭

#### F5 法遵伴侶 frame
- **核心敘事**：「我們陪你應對監理機關 / 通過認證」
- **客戶心裡買的**：監理 / 認證安心 + 經驗背書
- **適配 outcome**：GV-B / GV-C / GV-A 補強
- **適配窗口**：W3 CRO / W7 法遵長 / W1 CEO（法規危機）
- **適配觸發點**：T2 法規 / T3 危機 / T8 認證
- **失效模式**：客戶選四大會計師事務所 / 律所

#### F6 轉型陪跑夥伴 frame
- **核心敘事**：「我們陪你 3 年慢慢做」
- **客戶心裡買的**：長期 partner + 知識轉移 + 共同成長
- **適配 outcome**：AS-3 大型 / GV-C 全套
- **適配窗口**：W1 CEO / W2 CDO / W9 董事會
- **適配觸發點**：T1 高層自發
- **失效模式**：客戶不要長期承諾 / 預算每年要重審

#### F7 AI Lab Partner / 創新研發夥伴 frame
- **核心敘事**：「我們是你 AI 創新的研發夥伴，frontier-aligned」
- **客戶心裡買的**：frontier 認知 + 研發共創 + 同行 prestige
- **適配 outcome**：AS-3 frontier 包裝 / AS-2 frontier
- **適配窗口**：W1 CEO + W2 CDO（讀 frontier）
- **適配觸發點**：T1 高層自發 frontier 轉型
- **失效模式**：客戶覺得 IBM 不夠 frontier / 直接找 Anthropic / Palantir

#### F8 集團統合者 frame
- **核心敘事**：「我們幫你把分散 BU 拉成集團一致」
- **客戶心裡買的**：集團一致性 + 政治可行性 + 中立第三方
- **適配 outcome**：GV-C 集團版 / AS-2 集團藍圖
- **適配窗口**：W1 集團 CEO / W5 CFO / W10 PMO
- **適配觸發點**：T7 M&A / 集團整合
- **失效模式**：集團政治複雜化 / 子公司抵抗

#### F9 危機戰友 frame
- **核心敘事**：「我們是你 AI 出事的緊急 backup + 之後重建」
- **客戶心裡買的**：止血 + 經驗 + 信任的緊急號碼
- **適配 outcome**：GV-A 補強 / GV-B 補洞
- **適配窗口**：W1 CEO / W3 CRO
- **適配觸發點**：T3 危機
- **失效模式**：客戶選擇隱匿 / 公關處理 / 預算還在凍

### Frame 與 outcome 的關係

**同一 outcome 可走不同 Frame**，這是 Frame 維度的核心價值。範例：

GV-C 全套新建 + 落地包，依窗口不同走不同 Frame：
- 對玉山法遵長窗口 → **F5 法遵伴侶**：「我們陪您應對金管會」
- 對 YAGEO 集團 CFO 窗口 → **F8 集團統合者**：「我們幫您整合 12 個併購子公司治理」
- 對 CTBC frontier CEO 窗口 → **F7 AI Lab Partner**：「我們和您共建 AI-native 銀行的治理底盤」
- 對中型銀行 CIO 窗口 → **F1 工具供應商**：「watsonx.governance + Playbook 一站到位」

同樣 GV-C，4 個 Frame，4 種完全不同的 deck 開場 + 故事 + 投影片。**這就是「分層敘事庫」的真正樣子**——不只 Style 變，Frame 也變。
